数字宣言偏重 診断メモ

作成日: 2026-05-21 / 対象: episode_mixed モデルの数字宣言過多

結論

現在の数字宣言 90〜100% は、自然な最適方策としては扱わない。 分離済みNTP条件でも数字偏重が残ったため、相手指条件だけでは説明できない。 学習済みモデルは引き続き採用保留とし、再学習より前に教師生成・報酬・視点処理を見直す。

確認した事実

観点結果解釈
学習済みモデル separated_policy_report_episode_mixed.html で deterministic は全条件 数字宣言 100%。stochastic でも 90〜97%。 数字偏重は、NTPの指条件を分けても解消しない。
カウンター時の外し数字 初期局面で NUM(total=4, thumb=1) は、NTP指 0/1/2 のカウンターに対して常に外れる。 実装上、カウンターされた数字が外れると TP 側の手が下がるため、意図的な外し数字が勝ち筋になり得る。
有限深さsolver depth 1〜3 の初期局面では、数字だけに100%集中していない。depth 3 ではフェイント、スキップ、フラッシュも混ざる。 ルール木の浅い評価だけでは、学習済みモデルの数字100%を正当化しにくい。
BC教師生成 generate_bc_dataset と value iteration 由来の初期局面方策で NUM(total=0, thumb=0) 100% を確認。 BC教師データが数字偏重を初期化している疑いが強い。これは重要な赤信号。
RL環境の視点切替 exact solver は手番交代時に次状態価値の符号を反転する。一方、Gym環境は視点を入れ替えたobsを返し、PPOの将来報酬は通常の単一主体として蓄積される。 手番交代後の将来報酬が、直前の行動主体の価値として混ざる可能性がある。要検証。

現時点の原因候補

  1. BC教師の偏り: discounted value iteration / 状態列挙上限 / 未収束値 / fallback値により、初期局面の教師方策が数字100%になっている可能性。
  2. 外し数字の構造的価値: カウンター相手に、当たらない数字を宣言すると自分の手が下がるため、数字が「対カウンターもどき」として機能している。
  3. 視点切替とPPO目的の不一致: exact solver はゼロサムの符号反転を明示するが、PPO環境では手番交代後の報酬が同じエージェントの将来報酬として扱われる。
  4. material reward の補強: 手数内で手が下がる行動が小さく正に評価されるため、数字宣言の局所解を補強している可能性。

次に確認すべきこと

  • BC教師データの初期局面・代表局面における行動分布をHTML化し、数字偏重がどの段階で入るか確認する。
  • terminal-only と material の学習/評価を分け、material reward が数字偏重を増幅しているか確認する。
  • 手番交代時の視点切替に対し、PPOが受け取る将来報酬の符号が学習目的として妥当か検証する。
  • 外し数字を「フェイントの代替」として評価してよいのか、仕様・ゲーム性の観点で方針を決める。
この段階では再学習に進まない。まず、BC教師生成とRL環境の目的関数が、人間の意図する勝ち筋を学習させる形になっているかを確認する。
